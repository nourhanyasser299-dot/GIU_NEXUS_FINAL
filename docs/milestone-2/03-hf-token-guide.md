# Regenerating, Integrating, Demoing, and Shipping a Hugging Face API Token

This guide walks a developer in `HusseinSelim-1977/Nourhan_Project` through replacing a leaked or expired Hugging Face token, getting the three AI endpoints (`POST /api/ai/skills/extract`, `/api/ai/match`, `/api/ai/summarize`) running against it, demoing the result, and merging to `main`. Every command and path below is verified against the current `feature/hf-job-summarization` branch.

## Prerequisites

- Node ‚Č• 18, a clone of the repo, and `cd Backend && npm install` working.
- A Hugging Face account at https://huggingface.co with permission to manage tokens.

## Step 1 ‚ÄĒ Regenerate the token securely

Treat the old token as compromised. **Revoke first, mint second.**

- Open https://huggingface.co/settings/tokens.
- Click **Manage** on the old token ‚Üí **Invalidate and refresh**. Anything still using the old value starts getting `401`s ‚ÄĒ that is the intended signal.
- Click **+ New token**.
- **Type:** *Read* (the three endpoints only call `POST /models/...`; they never write).
- **Name:** `giu-nexus-backend` (or per-developer, e.g. `giu-nexus-alice`).
- **Copy the token once** into your password manager. Hugging Face will not show it again, and it must never appear in chat, tickets, screenshots, or commit messages.

## Step 2 ‚ÄĒ Integrate the token into the codebase

The backend reads the token from `process.env.HF_API_TOKEN` only ‚ÄĒ no hardcoded fallback. From `Backend/src/services/huggingface.client.js`:

```js
const token = process.env.HF_API_TOKEN || '';
if (token) headers.Authorization = `Bearer ${token}`;
```

Wire it in:

```bash
cd Backend
cp .env.example .env   # skip if .env already exists
```

Edit `Backend/.env` (model names already in `.env.example`):

```
HF_API_TOKEN=hf_your_new_token_here
HF_API_BASE=https://api-inference.huggingface.co
HF_SKILL_MODEL=jjzha/jobbert_skill_extraction
HF_MATCH_MODEL=sentence-transformers/all-MiniLM-L6-v2
HF_SUMMARY_MODEL=sshleifer/distilbart-cnn-12-6
```

`Backend/.gitignore` already lists `.env`, so the file stays out of git. Confirm before committing:

```bash
git status --ignored | grep -F .env   # .env must show as ignored
```

For deployments, inject the token through that platform's secret manager (Vercel, Render, Fly, GitHub Actions secrets, etc.) ‚ÄĒ never commit a `.env`.

## Step 3 ‚ÄĒ Run the demo with the new token

Open two terminals at the repo root.

```bash
# Terminal 1 ‚ÄĒ backend
cd Backend
npm install
npm test          # expect 41/41 passing
npm run dev       # boots Express on http://localhost:5000
```

```bash
# Terminal 2 ‚ÄĒ frontend (any static server is fine)
cd Frontend
npx http-server . -p 8080
```

Smoke-test from the shell that the token is actually reaching Hugging Face:

```bash
curl -s http://localhost:5000/api/ai/skills/extract \
  -H 'Content-Type: application/json' \
  -d '{"text":"Built React + Node.js dashboards with Postgres and AWS Lambda."}' | jq
```

A successful response has `"source": "huggingface"` and a non-empty `model` field. If you see `"source": "heuristic"`, the token did not reach the service ‚ÄĒ go back to Step 2.

Then drive the UI at http://localhost:8080:

1. Register or log in.
2. **Seeker dashboard** ‚Üí paste a CV ‚Üí **Extract Skills**. Status should read *"Generated by jjzha/jobbert_skill_extraction."*.
3. **Jobs board** ‚Üí paste the same CV ‚Üí **Score Jobs**. Each card grows a gold "X% Match" badge. Status names the MiniLM model.
4. Click a job card ‚Üí **Generate Summary**. Source line should read *"Generated by sshleifer/distilbart-cnn-12-6."*.

## Step 4 ‚ÄĒ Push the changes to `main`

Direct pushes to `main` are blocked by repo policy. Always go through a PR.

```bash
git checkout -b devin/$(date +%s)-hf-token-rotation
git add Backend/.env.example         # never .env itself
git commit -m "Rotate HF_API_TOKEN; refresh model defaults"
git push -u origin "$(git branch --show-current)"
```

Open a PR against `main` (`git_create_pr` in this environment, or the GitHub web UI), wait for the *Backend (lint + test + audit)* GitHub Actions check to go green, request a review, then squash-merge.

If the branch falls behind during review:

```bash
git fetch origin
git rebase origin/main          # resolve conflicts; tests should still be 41/41
git push --force-with-lease     # never plain --force
```

## Troubleshooting

- **401 Unauthorized from Hugging Face** ‚ÄĒ token typo or the new token lacks inference permission. Re-mint with type *Read*.
- **503 with `"Model is currently loading"`** ‚ÄĒ free-tier cold start. The client already passes `wait_for_model: true`; rerun the request after ~20 s.
- **`/api/ai/*` returns `source: "heuristic"` (or `jaccard` / `extractive`)** ‚ÄĒ the backend never saw `HF_API_TOKEN`. Confirm `Backend/.env` is in the same directory you ran `npm run dev` from.
- **Pre-commit blocks the push because `.env` was staged** ‚ÄĒ `git restore --staged Backend/.env` and recommit.

## Key takeaways

- **Revoke first, mint second.** A live old token is more dangerous than a brief outage.
- Only `Backend/.env` ever holds the literal token; `.env.example` documents the *names*.
- The three endpoints **fail open**: a missing or bad token degrades to a heuristic / Jaccard / extractive fallback, not a 500. Use that as a signal ‚ÄĒ if the UI status line says "fallback", you have a token problem.
- Every path to `main` goes through a PR with green CI. Force-pushes use `--force-with-lease`, never plain `--force`.
